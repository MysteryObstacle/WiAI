# Who is AI 对局界面设计规范 v1.0

## 1. 设计目标

`Who is AI` 是一款轻量社交推理网页游戏。对局界面必须让玩家快速理解当前阶段、随时追查其他玩家证据，并保持主页的黑白极简、点阵、终端感风格。

本版本不采用圆桌视角。默认母版是平面化三栏信息布局：

```text
顶部状态栏
左侧用户状态板 | 中间阶段主舞台 | 右侧用户档案
```

界面不应像后台管理系统，也不应像暖色桌游皮肤。它应该像一块克制的调查指挥台：黑色背景、低饱和灰阶、少量白色高亮、少量红色危险提示，信息密度清楚但不拥挤。

## 2. 页面母版

### 顶部状态栏

顶部状态栏回答：

- 现在是第几轮
- 当前是什么阶段
- 还剩多少秒
- 当前玩家是谁
- 流程走到哪一步

结构：

```text
左侧：Who is AI / 房间号 / 第 N 轮
中间：答题准备 ✓ / 答案揭晓 ✓ / 讨论 ✓ / 投票 ●
右侧：当前阶段 / 倒计时 / 你是 N 号用户
```

阶段轨道只表达流程状态，不切换服务器阶段。点击历史阶段时，后续可让右侧档案切到对应 tab，中间主舞台保持当前 phase。

### 左侧用户状态板

左侧是完整用户状态板，不再拆成左右两列。桌面端所有玩家都在左侧；中间主舞台和右侧档案不重复玩家列表。

每张用户卡显示：

- 用户编号和公开昵称
- 当前状态：等待、已提交、已公开、发言、已投票、冻结、托管
- 发言次数
- 被怀疑或被提及次数
- 当前票数
- 当前用户标签
- 当前选择目标或当前焦点状态

状态颜色语义：

- 普通用户：深灰卡片
- 当前用户：白色细边框 + “你”标签
- 当前选择/高疑似：红色细边框
- 已提交/已投票：绿色点或勾
- 等待中：黄色点
- 未开始/无状态：灰色点

左栏底部保留紧凑图例，解释绿/黄/红/灰状态。点击左侧用户只更新右侧档案，不改变当前阶段。

### 中间阶段主舞台

中间是页面核心。四个 stage 的中央 card 必须各有明显不同的内容结构，不能只是换标题。

阶段对应组件：

- `AnswerPrepCard`：问题卡 + 回答纸
- `AnswerRevealCard`：答案卡墙 + 当前答案详情
- `DiscussionCard`：焦点用户 + 相关消息 + 发言输入
- `VotingDecisionCard`：投票关系图 + 当前选择确认

原则：中间区域只突出当前阶段主任务，不重复左侧玩家状态板，也不重复右侧档案。

### 右侧用户档案

右侧是常驻用户档案，不是临时局势面板。默认跟随当前选中用户；投票阶段默认跟随当前投票目标。

Tabs：

- 玩家
- 答案
- 发言
- 投票
- 历史

右侧档案回答：

- 他答了什么
- 他说了什么
- 别人怎么质疑他
- 当前多少票
- 有哪些可疑标签或对比线索

答题准备阶段如果没有玩家证据，右侧展示本轮任务、回答提示、轮次信息，避免空面板。

## 3. Stage 1：答题准备

目标：玩家阅读问题，并提交一个具体、像人类的回答。该阶段安静、专注，其他信息弱化。

`AnswerPrepCard` 内容：

- 标题：准备回答
- 副标题：写一个足够具体的答案，证明你是人类。
- 本轮主题/问题 prompt
- 参考维度：具体经历、取舍理由、真实情绪、细节补充
- 答案纸：更大的深灰输入区，白色/语义边框，较大内边距
- 字数计数：`0 / 500`
- 主按钮：提交答案；提交后变为已提交
- 次级按钮：修改答案/撤回提交（沿用现有 cancel 行为）
- 提交统计：已提交 `N/M`

左侧用户状态重点显示提交状态，不显示身份。

右侧档案默认展示本轮任务、回答提示、当前轮次信息。

动效：

- 问题卡轻微淡入 + 上移
- 输入框 focus 边框变亮
- 提交后按钮文案和状态过渡
- 左侧用户卡出现已提交状态点/勾

## 4. Stage 2：答案揭晓

目标：公开所有用户答案，身份仍隐藏，玩家开始形成初步怀疑。

`AnswerRevealCard` 内容：

- 标题：答案揭晓
- 副标题：阅读每位用户的回答，标记你认为可疑的地方。
- 答案卡墙：一张卡代表一名用户答案
- 当前答案详情：展示当前选中用户完整答案
- 可疑标签：像模板、缺少细节、过于理性、真实
- 上一条/下一条切换

答案卡显示：

- 答案编号或用户编号
- 答案摘要
- 字数
- 可疑标签数量
- 是否已查看/当前选中

右侧档案跟随当前答案用户，默认高亮“答案”tab。

动效：

- 答案卡依次出现
- 当前选中答案卡轻微亮边
- 标签贴上时小弹性
- 切换答案时内容横向或透明度过渡

## 5. Stage 3：讨论

目标：围绕答案进行质疑、辩护、诱导和反驳。讨论阶段是游戏核心。

`DiscussionCard` 内容：

- 标题：讨论阶段
- 副标题：围绕答案提出质疑，或为自己辩护。
- 当前焦点用户
- 焦点摘要：被质疑次数、可疑标签、最近行为
- 相关发言：优先展示与焦点用户相关的消息；无相关消息时显示完整近期讨论
- 发言输入
- 快捷回应：同意、质疑、追问、举例

讨论消息分两层：

- 中间主舞台：当前焦点相关消息
- 右侧档案：完整用户发言记录和上下文

左侧用户状态重点显示发言次数、被提及次数、怀疑热度、是否当前焦点。

右侧档案默认显示当前焦点用户，并高亮“发言”tab。

动效：

- 新消息淡入并轻微上移
- 被提及用户卡片闪一下
- 当前焦点用户卡片白色或红色细边
- 快捷回应 hover 轻微亮起
- 倒计时最后 10 秒轻微脉冲

## 6. Stage 4：投票

目标：选择最像 AI 的对象并确认投票。中间面板不能重复左侧用户列表，而应使用更游戏化的投票关系图。

`VotingDecisionCard` 内容：

- 标题：最终投票
- 副标题：选择你认为最像 AI 的用户。不能投给自己。
- 平面投票桌/投票关系图
- 用户圆点围绕抽象 X 形桌面排列
- 箭头从投票者指向被投者
- 当前选择摘要
- 确认投票按钮

投票图规则：

- 白色圆：普通用户
- 红色圆：当前选择
- 绿色勾：已投票
- 灰色空心：未投票
- 黄色边框：等待中
- 用户圆旁显示当前票数
- 投向采用模式 A：公开投向，显示谁投了谁

左侧用户状态只显示全局投票状态：你/未投/已投/当前选择。

右侧档案默认显示当前选择对象，并高亮“投票”tab。

动效：

- 点击圆点后目标变红并轻微放大
- 确认后从当前用户圆点飞出箭头指向目标
- 票数数字弹跳
- 被投目标外圈短暂闪烁
- 已投用户出现勾号

## 8. 数据映射

现有 `WiaiSnapshot` 是主数据源。需要通过 view-model helpers 派生 UI 状态：

- 当前轮 answers/messages/ballots
- 每位玩家提交状态
- 每位玩家发言次数
- 每位玩家被提及次数（第一版可用消息文本包含编号的简单检测）
- 每位玩家被投票数
- 当前玩家自己的 ballot
- 当前选择对象
- 公开显示名，禁止暴露 debug AI 名称
- 阶段默认档案 tab

投票关系图需要：

- `actorSessionPlayerId`
- `targetSessionPlayerId`
- `abstain`
- 玩家编号到节点坐标的映射
- 当前用户节点
- 当前选择目标节点

## 9. 视觉规范

颜色以 shadcn semantic token 为主，必要时映射到以下暗色语义：

- 页面背景：`#050505`
- 主面板背景：`#111111`
- 卡片背景：`#171717`
- 次级卡片：`#202020`
- 边框：`#2F2F2F`
- 弱文字：`#9CA3AF`
- 主文字：`#F5F5F5`
- 主按钮：白底黑字
- 危险/高疑似：`#EF4444`
- 成功/已完成：`#22C55E`
- 等待/中立：`#EAB308`

圆角：

- 页面大面板：`12px`
- 状态卡/阶段卡内部块：`8px`
- 小标签/状态点容器：使用 shadcn 默认 radius

间距：

- 页面外边距：`20px`
- 三栏 gap：`16px`
- 面板内部主要 gap：`16px`
- 紧凑信息行 gap：`8px`

字体层级：

- 页面标题：20-24px，粗体
- 阶段标题：22-28px，粗体
- 卡片标题：16-18px，粗体
- 正文：14-16px
- 辅助文字：12-13px
- 状态标签：11-12px

## 10. 动画规范

动效应轻量、克制，符合黑白极简风格。

通用：

- 阶段切换：opacity + 可选 translateY，180-240ms
- 卡片 hover：边框变亮，120ms
- 当前选择：边框高亮 + scale 1.02
- 按钮点击：scale 0.98 后恢复

GSAP 使用边界：

- 只用于阶段切换、答案揭晓、消息出现、投票确认
- 使用 `autoAlpha`、`x/y`、`scale`
- 必须 `gsap.context()` cleanup
- 必须 `clearProps`
- 必须尊重 `prefers-reduced-motion`

不要用 GSAP 替代 shadcn hover/focus/selected/disabled 状态。

## 11. 响应式

桌面端：

```text
左栏 240-260px
中间自适应
右栏 320px
```

平板端：

- 左栏 200px
- 中间自适应
- 右侧档案可变抽屉或压缩为下方/右侧面板

手机端：

```text
顶部状态栏
用户横向条
当前阶段主卡
底部快捷入口
调查档案抽屉
```

移动端只显示当前阶段主任务，其他数据通过 drawer 查询。

## 12. 关键交互规则

选择用户：

- 点击左侧用户：只更新右侧档案，不改变主舞台
- 点击投票图用户：设置投票目标，并让右侧档案跟随该目标
- 点击右侧档案相关用户：更新 selectedPlayerId

阶段导航：

- 顶部阶段轨道不是手动切换阶段入口
- 点击历史阶段只改变右侧档案 tab 或历史内容，中间仍保持当前 phase

投票确认：

- 第一步选择用户
- 第二步确认投票
- 确认后按钮变为已投票，目标锁定，图中显示箭头，左侧状态更新

## 13. 开发优先级

P0：结构稳定。

- 三栏母版固定高度
- 左侧完整用户状态板
- 中间阶段主舞台
- 右侧常驻档案

P1：四个阶段卡差异化。

- 答题准备卡
- 答案揭晓卡
- 讨论卡
- 投票关系图卡

P2：交互。

- selectedPlayerId
- selectedAnswerPlayerId
- selectedVoteTargetId
- right panel tabs
- phase nav history behavior

P3：动画。

- 阶段切换
- 答案卡出现
- 消息出现
- 投票箭头/票数弹跳

P4：移动端。

- 用户横条
- 主阶段卡
- 调查档案 drawer
- 底部快捷入口

## 14. 验收标准

1. 四个 stage 的中央主卡视觉和内容明显不同。
2. 左侧用户栏能表达提交、发言、投票、怀疑状态。
3. 右侧用户档案始终跟随当前选中用户或当前投票目标。
4. 投票阶段中间不重复用户列表，而使用投票关系图。
5. 用户能从任意阶段查看答案、发言、票数和历史信息。
6. 当前阶段主任务始终是页面最强视觉焦点。
7. 动画只用于阶段切换、答案揭晓、消息出现、投票确认等关键节点。
8. 整体风格与主页黑白极简、点阵、终端感保持一致。
